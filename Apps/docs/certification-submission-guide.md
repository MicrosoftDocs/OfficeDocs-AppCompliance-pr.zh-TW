## <a name="introduction"></a>簡介

Microsoft 365 應用程式合規性計畫是 Microsoft 365 應用程式合規性計畫的一部分，可為企業組織提供保證和信賴，讓企業組織在將協力廠商開發人員應用程式/增益集整合到 Microsoft 365 平臺時，能夠充分保護資料和隱私權。 通過驗證的應用程式和增益集將在整個 **Microsoft 365** 生態系統中指定為 Microsoft 365 認證。 

藉由參與 Microsoft 365 認證計畫，您同意這些補充條款，並遵守適用于您參與 Microsoft 365 認證計畫的任何隨附檔，並搭配 Microsoft Corporation (「Microsoft」、「we」、「us」 或 「our」) 。 您代表並保證您有權代表您自己、公司及/或其他實體，視情況接受這些 Microsoft 365 認證補充條款。 我們隨時可能會變更、修改或終止這些補充條款。 在任何變更或增修之後，您繼續參與 Microsoft 365 認證計畫，表示您同意新的補充條款。 如果您不同意新的補充條款，或我們終止這些補充條款，您必須停止參與 Microsoft 365 認證計畫。

本檔的目標是 ISV (獨立軟體廠商) 提供 Microsoft 365 認證程式的相關資訊、啟動程式的必要條件，以及 ISV 必須具備的特定安全性控制詳細資料。  您可以在 Microsoft 365 應用程式合規性計畫 [頁面](../overview.md)下找到 Microsoft 365 應用程式合規性計畫的一般資訊。 

> [!IMPORTANT]
> 目前，Microsoft 365 認證適用于所有專案：
>* Microsoft Teams 應用程式 (索引標籤、Bot 等) 。
>* Sharepoint Apps/增益集
>* Office 增益集 (Word、Excel、PowerPoint、Outlook、Project、OneNote) 
>* WebApps

## <a name="prerequisites"></a>必要條件

### <a name="publisher-attestation"></a>發行者證明

在獲得 Microsoft 365 認證程式之前，您必須已完成發行者證明。 不過，您可以在完成發行者證明之前，先啟動 Microsoft 365 認證程式。  

### <a name="read-the-microsoft-365-certification-specification"></a>閱讀 Microsoft 365 認證規格

Microsoft 建議所有 ISV (獨立軟體廠商) 完整閱讀此 Microsoft 365 認證規格，以確保範圍內的環境和應用程式/增益集符合所有適用的控制項。 這有助於確保評定程式順暢。

## <a name="microsoft-365-certification-specification-updates"></a>Microsoft 365 認證規格更新 

更新 Microsoft 365 認證規格預期大約每六到十二個月。 這些更新可能會引進新的目標安全性網域和/ 或安全性控制。 更新將以開發人員的意見反應、威脅環境的變更為基礎，並在程式成熟時增加其安全性基準。 

已啟動 Microsoft 365 認證評量的 ISV 可以使用評估啟動時有效的 Microsoft 365 認證規格版本繼續評估。 所有新的提交，包括年度重新認證，都必須根據發佈的版本進行評估。

> [!NOTE]
> 您不需要遵守此 Microsoft 365 認證規格內的所有控制項，即可獲得認證。 不過，針對此 Microsoft 365 認證規格中所討論的每個安全性網域， (傳遞不會在) 公開的閾值。 某些控制項會分類為「**硬性失敗**」，這表示缺少這些安全性控制項會導致評定失敗。 

## <a name="certification-scope"></a>認證範圍

範圍內 **環境** 是支援傳遞應用程式/增益集程式碼的環境，並支援應用程式/增益集可能與之通訊的任何後端系統。 除非有足夠的區隔，否則任何額外的已連線環境也會包含在範圍內，而且連線到的環境不會影響範圍內環境的安全性。 任何災害復原環境也都必須包含在評量範圍內，因為如果主要環境發生任何狀況，這些環境就必須履行服務。  範圍  **內系統元件**   一詞會參考範圍內環境中使用 **的所有** 裝置和系統。 範圍內的元件包括但不限於：
* Web 應用程式 () 。
* 伺服器。
* 防火牆 (或對等) 。
* 開關。
* 負載平衡器。
* 虛擬基礎結構。
* 雲端提供者 Web 管理入口網站 

> [!IMPORTANT]
> 範圍內的環境必須有 DMZ，而且應用程式/增益集的支援環境必須與內部商務系統和公司環境區隔，因此只會將評定活動的範圍限制在範圍內的系統。 認證分析師會在評定期間驗證分割技術，以及檢閱滲透測試報告，其中應該包含測試，以驗證使用中任何分割技術的有效性。

### <a name="infrastructure-as-a-service-iaas-platform-as-a-service-paas-and-software-as-a-service-saas"></a>基礎結構即服務 (IaaS) 、平臺即服務 (PaaS) 和軟體即服務 (SaaS)  
在 IaaS 和/或 PaaS 用來支援受檢閱之應用程式或增益集程式碼傳遞的基礎結構時，雲端平臺提供者將負責在整個認證程式中評估的一些安全性控制措施。 因此，雲端平臺提供者必須透過外部合規性報告，例如 [PCI DSS] 證明合規性 (AOC) 、ISO27001 或 [SOC 2] 類型 II 報告，為認證分析師提供安全性最佳做法的獨立外部驗證。 

附錄 F 會根據下列部署類型，以及應用程式/增益集是否要外流 M365 資料，提供哪些安全性控制項可能適用的詳細資料： 
* ISV 裝載 
* 裝載的 IaaS 
* 裝載的 PaaS/無伺服器 
* 混合式託管 
* 共用託管 

部署 IaaS 或 PaaS 時，您必須提供裝載在這些部署類型內之環境的辨識項。

### <a name="sampling"></a>採樣

支援認證評量的辨識項要求，應以考慮不同作業系統、裝置的主要功能和不同裝置類型的範圍內系統元件範例為基礎。 在認證程式開始時，將會選取適當的範例。 下表應作為範例大小可能的指南：

|母體大小              | 範例                  |
|---------------------------- |-------------------------|
|<5|1|
|>5 & <10|2|
|>9 & <25|3|
|>24|4 |

> [!NOTE]
>如果在初始範例中包含的裝置之間發現不一致，則在評估期間可能會增加樣本大小。 

## <a name="certification-process"></a>認證程式

開始認證程式之前，您必須已成功完成發行者證明。 完成之後，您的 Microsoft 365 認證程式將會繼續進行，如下所示：

### <a name="preparation"></a>準備
1. 流覽至合作夥伴中心，並檢閱您已完成 [的發行者證明](../docs/attestation.md) 檔。 如有必要，您可以編輯和更新回應;不過，如果您這樣做，您必須重新提交證明檔以供核准。 如果您的提交時間超過三個月，我們將要求您重新提交發行者證明以進行檢閱和驗證。 
1. 請仔細閱讀 [Microsoft 365 認證提交指南](../docs/certification-submission-guide.md) ，以瞭解您需要什麼。 請確定您能夠滿足 Microsoft 365 認證提交指南中所指定的 [控制](../docs/certification-submission-guide.md#app-certification-criteria) 需求。
1. 在合作夥伴中心內，按一下 [開始認證]。 這會帶您前往初始檔提交入口網站。 提交 [您的初始檔提交](../docs/certification-submission-guide.md#initial-document-submission)。 這可協助我們根據您的應用程式架構和處理客戶資料的方式，判斷評量的範圍為何。 請經常檢查此頁面，以查看您的提交是否已接受。

>[!NOTE]
>對於所有 Office 應用程式，您可以參考我們的 [Office Apps 使用者指南](../docs/userguide.md)。 對於所有 WebApp，您可以參考我們的 [SaaS 應用程式使用者指南](../docs/saasuserguide.md)。

### <a name="assessment"></a>評定
1. 一旦接受您的初始檔提交，您應用程式所需的一組安全性控制項將會自動顯示在入口網站中。 接著，您必須提交每個控制項的辨識項，以證明控制項已就緒。 請記住，您有 **60 天** 的時間提交所有辨識項。 分析師會檢閱您的辨識項，並核准控制項或要求新的或額外的辨識項。 請經常檢查此頁面，以查看是否已接受您的辨識項。
### <a name="certification"></a>認證
1. 一旦分析師驗證您的提交，您將會收到認證決策的通知。 獲得認證的應用程式將會在其 **應用程式的 AppSource** 和 **Microsoft 檔** 頁面上收到徽章。 您可以 [在這裡](../docs/enterprise-app-certification-guide.md#program-benefits)閱讀認證的完整優點。

## <a name="review-and-re-certification"></a>檢閱並重新認證
如果您的應用程式在任何時間點進行 [重大變更](../docs/certification-submission-guide.md#significant-changes) ，您都必須通知我們。

您也必須每年進行重新認證。 這需要針對您目前的環境重新驗證範圍內的控制項。 此程式最多可在認證到期前 90 天開始。 您現有的認證不會在重新認證期間過期。 所有方案的重新認證會在您的 Microsoft 365 認證一周年到期。

如果您的認證未在到期日之前更新，您的應用程式認證狀態將會被撤銷。 所有徽章、圖示和相關聯的認證商標都會從您的應用程式中移除，而且您將禁止將您的應用程式公告為 Microsoft 365 認證。


> [!IMPORTANT]
> **提交時間範圍：** 您預期評定程式平均應該需要 30 天的時間，前提是您能夠經常檢查提交狀態，並及時回應批註和補充辨識項要求。 開始認證程式時，最多允許 60 天完成評定。 如果所有提交尚未在 60 天的期間內完成，提交將會發出失敗，而且程式必須重新開始。 這些結果不會公開。


## <a name="initial-document-submission"></a>初始檔提交


初始檔提交將協助認證分析師執行範圍設定，並判斷您的評量範圍中將包含哪些專案。 之後，您必須提交用來執行評估的支援檔和辨識項。 您的初始提交必須包含下列指定的資訊。 如需其他指南，請參閱 [初始檔子委任指南](../docs/initialdocumentsubguide.md)。

| **檔 &nbsp; 概觀**     |   **檔詳細資料**  |
| -------------------------| -----------------------------|
|**應用程式/增益集描述** | 應用程式/增益集用途和功能的描述。 這應該可讓認證分析師充分瞭解應用程式/增益集的功能及其用途
|**滲透測試報告** |滲透測試報告已在過去 12 個月內完成。 此報告必須包含支援部署應用程式/新增的環境，以及支援應用程式/增益集作業的任何其他環境。 **注意：** 如果您沒有執行年度滲透測試，您可以選擇透過認證程式來完成這些測試。|
|**架構圖表**|邏輯架構圖，代表應用程式/增益集支援基礎結構的高階概觀。 這必須包含 **所有** 裝載環境和支援支援應用程式/增益集的基礎結構。 此圖表必須描述環境中所有不同的支援系統元件，以協助認證分析師瞭解範圍內的系統，並協助判斷取樣。 也請指出使用何種裝載環境類型;ISV Hosted、IaaS、PaaS 或混合式。 **注意：** 使用 SaaS 的位置，請指出用來在環境中提供支援服務的各種 SaaS 服務。|
|**公用使用量** | 詳細說明支援基礎結構使用 **的所有** 公用 IP 位址和 URL。 這必須包含配置給環境的完整可路由 IP 範圍，除非已實作適當的分割來分割使用中的範圍 (需要適當的分割辨識項) |
|**資料流程圖** |詳細說明下列各項的流程圖：
||&#x2713; M365 資料流程向應用程式/增益集 (，包括 [EUII](#euii) 和 [OII](#oii) ) 。|
||&#x2713;適用之基礎結構 (內的 M365 資料流程) |
||&#x2713;圖醒目提示儲存資料的位置和內容、如何將資料傳遞給外部協力廠商 (包括協力廠商) 的詳細資料，以及如何透過開放/公用網路和待用網路保護傳輸中的資料。|
|**API 端點詳細資料**| 應用程式所使用之所有 API 端點的完整清單。 若要協助瞭解環境範圍，請在您的環境中提供 API 端點位置。                                
|**Microsoft API 許可權**| 提供檔，詳細說明 **所有** 使用的 Microsoft API，以及應用程式/增益集要運作的要求許可權，以及要求許可權的理由|
|**資料儲存體類型** |描述下列專案的資料儲存和處理檔：|
||&#x2713;接收和儲存客戶 M365 資料 [EUII](#euii) 和 [OII](#oii) 的範圍|
||&#x2713;資料保留期間。|
||&#x2713;擷取客戶 M365 資料的原因。|
||&#x2713;客戶 M365 資料的儲存位置 (應包含在上述) 提供的資料流程圖中。|
|**合規性確認**|支援發行者證明提交中包含的外部安全性架構檔，或在檢閱 Microsoft 365 認證控制項時考慮的檔。 目前支援下列三項：|
||&#x2713; AOC) 合規性 ([PCI DSS](#pci-dss) 證明。|
||&#x2713; [SOC 2](#soc-2) 類型 I/類型 II 報告。|
||&#x2713; [ISMS](#isms)  /  [IEC](#iec) - 1S0/IEC 27001 適用性聲明 (SoA) 和認證。|
|**Web 相依性**|列出應用程式/增益集與目前執行中版本使用之所有相依性的檔。|
|**軟體清查**|最新的軟體清查，其中包含在範圍內環境中使用的所有軟體以及版本。|
|**硬體清查**| 支援基礎結構所使用的最新硬體清查。 這會用來協助您在執行評定階段時進行取樣。 如果您的環境包含 PaaS，請提供所取用服務的詳細資料。|

## <a name="evidence-collection-and-assessment-activities"></a>辨識項收集和評量活動

認證分析師必須在定義的範例集內檢閱所有系統元件的辨識項。 支援評估程式所需的辨識項類型包括下列任何或所有專案：

**Evidence 集合**

* 在上方的初始檔提交一節中反白顯示[的初始檔](#initial-document-submission) 
* 原則檔 
* 處理檔 
* 系統組態設定 
* 變更票證 
* 變更控制項記錄 
* 系統報告

將使用各種方法來收集完成評定程式所需的辨識項。  此辨識項集合的格式可能如下： 
* 文件 
* 螢幕擷取畫面 
* 採訪 
* 螢幕共用 

在評估程式期間，將會決定所使用的辨識項集合技術。 如需提交中所需辨識項類型的具體範例，請參閱范 [例辨識項指南](../docs/certification-sample-evidence-guide.md)。

**評定活動**

認證分析師會檢閱您提供的辨識項，以判斷您是否已充分符合此 Microsoft 365 認證規格內的控制項。 

可能的話，為了減少完成評定所需的時間，應該事先提供 [初始檔提交](#initial-document-submission)   中詳述的任何或所有檔。

認證分析師會先檢閱初始檔提交所提供的辨識項和發行者證明信息，以識別適當的查詢行、取樣大小，以及如上所述取得進一步辨識項的需求。  認證分析師會分析收集到的所有資訊，以得出您如何符合此 Microsoft 365 認證規格內控制項的結論。 

## <a name="app-certification-criteria"></a>應用程式認證準則

您的應用程式、支援基礎結構和支援的檔將跨下列安全性網域進行評估：

1. [**Application Security**](#application-security)
1. [**作業安全性/安全部署**](#operational-security)
1. [**資料處理安全性和隱私權**](#data-handling-security-and-privacy)

這些安全性網域都包含特定的主要控制項，其中包含一或多個將在評定程式中評估的特定需求。 為確保所有大小的開發人員都具備 Microsoft 365 認證，系統會使用評分系統來評估每個安全性網域，以判斷每個網域的整體分數。 每個 Microsoft 365 認證控制項的分數會根據該控制項未符合的認知風險，在 1 個 (低) 和 3 個 (高) 之間配置。 每個安全性網域都有要視為通過的最小百分比標記。 此規格的某些元素包含一些自動失敗準則：

- API 許可權未遵循最低許可權原則 (PoLP) 。  
- 不需要滲透測試報告。
- 沒有反惡意程式碼防禦
- Multi-Factor Authentication 不會用來保護系統管理存取權。  
- 沒有修補程式。  
- 沒有適當的 [GDPR](#gdpr) 隱私權注意事項。  

## <a name="application-security"></a>Application Security

應用程式安全性網域著重于下列三個領域： 
* GraphAPI 許可權驗證 
* 外部連線檢查
* 應用程式安全性測試 

### <a name="graphapi-permission-validation"></a>GraphAPI 許可權驗證

執行 GraphAPI 許可權驗證，以驗證應用程式/增益集不會要求過度寬鬆許可權。 這是透過手動檢查要求的許可權來執行。 認證分析師會針對發行者證明提交交叉參考這些檢查，並評估所要求的存取層級，以確保符合「最低許可權」做法。 在認證分析師認為不符合這些「最低許可權」做法的情況下，認證分析師會與您公開討論，以驗證所要求許可權的業務理由。 在此檢閱期間找到的任何發行者證明提交不一致，也會取得意見反應，以便更新您的發行者證明。 

### <a name="external-connectivity-checks"></a>外部連線檢查

在評估過程中，分析師會執行應用程式功能的簡要逐步解說，以識別 M365 外部的連線。  任何未識別為 Microsoft 或直接連線到您服務的連線，都會在評量期間標示並討論。

### <a name="application-security-testing"></a>應用程式安全性測試

適當地檢閱與您的應用程式/增益集和支援環境相關聯的風險，對於為客戶提供應用程式/增益集安全性的保證至關重要。 如果您的應用程式與 Microsoft 未發佈的任何服務有任何連線，則必須執行滲透測試形式的應用程式安全性測試。 如果您的應用程式在沒有連線到任何非 Microsoft 服務或後端的情況下獨立運作，則不需要滲透測試。
